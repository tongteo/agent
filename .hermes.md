# OpenRouter Agent CLI

AI Agent CLI built with Node.js — terminal-based agent with tool integration, LSP support, and Playwright-based web providers (Gemini/Claude).

**When working with this project**, follow these conventions and instructions below.

## Code Conventions

- **Language**: JavaScript (Node.js >= 18), CommonJS (`require`/`module.exports`)
- **Formatting**: 2-space indent, semicolons required, single quotes preferred
- **Comments**: JSDoc `/** ... */` for all exported functions and classes
- **Naming**: `camelCase` for variables/functions, `PascalCase` for classes
- **Error handling**: try/catch with meaningful messages; tool calls return `Error:` prefix strings on failure, never throw up to the loop
- **File organization**: one class / concern per file under `src/core/tools/`

## Tool Patterns

The agent uses an XML-like tool call format (`<tool>name</tool><params>{...}</params>`), with fallback support for JSON-format (`tool_name\n{...}`) and `longcat` format.

**Adding a new tool:**
1. Create the implementation in `src/core/tools/<name>.js`
2. Export from `src/core/tools/index.js` (ToolRegistry)
3. Add the tool name to `ToolParser.TOOL_NAMES` in `src/core/agent.js`
4. Write tests in `tests/test-<name>.js`

## Skills System (NEW)

The agent now has a Hermes-compatible skills system. Skills are `.md` files
with optional YAML frontmatter stored in `skills/`.

**Tools available to the model:**
- `skills_list` — browse available skills (supports optional `filter`)
- `skill_view` — read a skill's full content by name
- `skill_load` — inject a skill's instructions into the system prompt
- `skill_search` — full-text search across all skill names, tags, descriptions, and bodies
- `skill_manage` — create, edit (replace), patch (find-and-replace), or delete skills

**How it works:**
1. Skills live in `nodejs/skills/<name>.md`
2. Each file may have `---` YAML frontmatter (name, description, tags)
3. `skill_load` injects the body into `[Loaded Skills]` section of the prompt
4. The injected content is re-applied on every turn via `MessageHandler.getSystemContext()`
5. Skills persist for the session but are not loaded automatically — the model must explicitly call `skill_load`
6. The model can CREATE new skills with `skill_manage(action:"create")` after discovering useful patterns
7. The model can PATCH existing skills with `skill_manage(action:"patch")` to update knowledge

**SkillManager** (`src/core/skills.js`):
- `list()`, `view(name)`, `load(name)` — browse and activate
- `search(query)` — full-text ranking with score + context snippets
- `create(name, body, description, tags)` — new skill file
- `edit(name, newContent)` — replace entire skill content
- `patch(name, oldStr, newStr)` — find-and-replace within a skill
- `delete(name)` — remove skill file
- `getLoadedPromptAdditions()` — formatted string for system prompt injection

## Provider Architecture

Two web-based providers using Playwright + CDP:
- **Gemini Web** (`src/models/gemini-cookies.js` + `src/bridges/gemini-client.js`)
- **Claude Web** (`src/models/claude-cookies.js` + `src/bridges/claude-client.js`)

Both share a common `BrowserManager` (`src/bridges/browser-manager.js`) for CDP connections.

The agent also supports native function calling via OpenRouter-compatible API —
when `this.model.pendingToolCalls` is populated, the loop uses native `tool_call_id` role alternation instead of the XML parser.

## Key Architecture Rules

1. **Sequential tool execution** — tools are run one-at-a-time (no parallel), because later tools often depend on side effects of earlier ones (write_file → execute).
2. **Agent loop** — up to 20 iterations per turn, alternating model response → tool execution → model response.
3. **Context trimming** — `ContextManager` estimates tokens and auto-trims history when approaching the model's context limit.
4. **Session persistence** — working directory + env vars survive across sessions.
5. **Security validator** — dangerous commands (`rm -rf`, `mkfs`, `dd`, fork bombs) get flagged before execution.
6. **Subagent delegation** — `SubagentManager` spawns parallel child agents (used via `use_subagent` tool).

## Prompt Design

- **Compact prompt** (`getCompactPrompt`): for models with limited context (Gemini Web) — 14 tools max, minimal instructions
- **Full prompt** (`getSystemPrompt`): for capable models — detailed rules, multiple examples
- **Intent parser fallback** (`IntentParser`): regex-based natural-language-to-tool dispatch for models that don't follow XML format

## Development Workflow

1. Understand the tool → read the relevant source file
2. Run the test suite before making changes: `cd nodejs && npm test`
3. After changes, run `npm test` again — all 106+ tests must pass
4. For LSP changes, verify with the actual LSP connection (requires a language server)
5. Keep `README.md` in sync with architecture changes
6. Avoid changing the tool XML format — it's the primary contract with the model

## Cross-Platform

- **Linux** is the primary target (CI runs on Linux)
- Tests should remain platform-agnostic (the validator handles cross-platform)
- The agent auto-detects OS for shell command hints in the prompt

## Common Pitfalls

- After splitting tools.js into modules, all tools are still flat-registered in ToolRegistry — do not re-introduce a single monolithic tools.js
- Changing the XML tool format requires updating both `ToolParser.parse()` AND the system prompt examples in `AgentPrompt`
- The gemini-web provider sometimes returns HTML-escaped XML (`&lt;tool&gt;`) — `ToolParser` handles unescaping
- On first run, the project needs either `GEMINI_COOKIES=1` or `CLAUDE_COOKIES=1` in `.env` to function
